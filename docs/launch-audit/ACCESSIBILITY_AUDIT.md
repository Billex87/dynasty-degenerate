# Accessibility Audit – Dynasty Degens

This audit identifies accessibility risks observed in the desktop version of the Skids Get Beat league.  Because of limited testing, keyboard navigation and screen reader behaviour could not be fully verified.  The findings below focus on visible issues that impede users with disabilities.

| ID | Severity | User‑impact category | Location | Problem | Why it matters | Recommended fix | Acceptance criteria | Launch priority |
|---|---|---|---|---|---|---|---|---|
| **A11y‑001** | High | Navigation | Site‑wide | **Lack of keyboard focus indicators.**  There are no visible outlines or state changes when tabbing through links or buttons. | Users who navigate via keyboard (including those with motor impairments or screen reader users) may not know which element is focused. | Implement visible focus styles (e.g., outline or underline) on all interactive elements.  Ensure that focus order follows a logical sequence. | Users can tab through the page and see which element is active at all times. | Must fix |
| **A11y‑002** | High | Interaction/Control | League switching | **Switch League button is non‑functional**.  This also affects accessibility; keyboard users cannot change leagues at all. | Prevents access to other content, particularly for users relying on the button rather than the `Find Leagues` page. | Fix the underlying bug and ensure the button is accessible (proper `<button>` element, labelled, focusable). | Users can activate the button with keyboard or mouse and open the league selection modal. | Must fix |
| **A11y‑003** | High | Colour contrast | Tables, nav bars | **Low contrast text** – Many labels and values are dark grey on dark backgrounds; red and green value change badges rely solely on colour. | Users with low vision or colour‑blindness may not be able to read the text or interpret value changes. | Increase contrast ratios to meet WCAG AA (4.5:1 for normal text) and AAA for smaller text.  Add icons or labels to value change indicators. | All text has sufficient contrast; value change conveys direction via icon and label in addition to colour. | Should fix |
| **A11y‑004** | Medium | Labelling | Player modal & table | **Missing ARIA labels and alt text** – Icons, avatars and buttons lack alt attributes; the player modal’s close function has no label because there is no close button. | Screen reader users cannot understand the purpose of these controls. | Add `aria-label` attributes to icon‑only buttons; include `alt` attributes for avatars; provide a labelled close button in modals. | Screen readers read meaningful labels for all controls. | Should fix |
| **A11y‑005** | Medium | Forms & inputs | Landing page | **Input fields lack associated labels** – The Sleeper username and league ID fields rely on placeholder text instead of labels. | Placeholders disappear when users type and are not read by screen readers as labels. | Add `<label>` elements associated with each input and use visible labels above or inside the fields. | Inputs are clearly labelled and accessible via assistive technology. | Should fix |
| **A11y‑006** | Medium | Focus management | Player modal | **Focus not trapped in modals.**  When the player modal opens, focus may remain on elements behind the modal. | Users navigating with screen readers or keyboards may inadvertently interact with the background page. | Implement focus trapping in modals: move focus to the modal when it opens and return focus to the triggering element when closed. | Opening the modal shifts focus inside; tabbing cycles within the modal until it is closed. | Should fix |
| **A11y‑007** | Low | Animations | Loading overlay | **Unclear screen reader announcement** – The loading overlay contains humorous text but no ARIA live region or progress announcements. | Users with visual impairments might not know that content is loading or stuck. | Add `role="status"` or `aria-live` attributes to the overlay.  Provide serious progress messages along with humour. | Screen readers announce “Loading league data” and update progress. | Nice to have |
| **A11y‑008** | Low | Touch targets | Mobile untested | Buttons and toggles appear small; on mobile they may fall below the recommended 44×44 px. | Harder to tap for users with motor impairments. | Increase button sizes and spacing on small screens; ensure tap targets meet guidelines. | Buttons are at least 44 px high/wide on mobile breakpoints. | Nice to have |

### Additional considerations

- Test with screen reader software (e.g., NVDA or VoiceOver) to ensure all text is announced in a logical order and that content hidden from visual users is not announced unnecessarily.
- Provide `skip to content` links at the top of the page to allow keyboard users to bypass repetitive nav bars.
- Ensure all charts and graphs have accessible alternatives (e.g., data tables or text summaries).

Addressing these accessibility gaps is crucial for launching an inclusive product.  Some fixes (e.g., focus indicators, button labels) are straightforward and should be prioritised for launch.
